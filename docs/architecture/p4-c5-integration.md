# P4 与 C5 联动

适合谁看：
- 准备做双端联调的人
- 准备改动 P4 / C5 边界行为的人

读完会得到什么：
- 知道 P4 和 C5 是如何协作的
- 知道联调时应该先验证哪一侧
- 知道边界改动为什么更需要文档先行

## 先说清楚“联动”到底指什么

这里说的联动，不是两边代码互相调用函数。它真正的意思是：

- C5 把无线能力提供出来
- P4 使用这份无线能力
- P4 再负责把网络侧和 UART 侧组织成完整桥接链路

也就是说，C5 更靠近“能力提供”，P4 更靠近“业务编排”。

## 这条协作链怎么理解

如果把整个系统看成一条链路，可以把它分成三段：

1. 外部设备和无线链路
2. P4 内部桥接
3. 飞控串口链路

其中第一段更依赖 C5，第二段主要在 P4，第三段则由 P4 直接面对飞控。

结合当前实现，再补两条容易搞混的事实：

- P4 的主桥接链路在启动时就会拉起默认 `UDP 8888` / `TCP 8889` 监听
- `net_*` 命令对应的是另一套独立网络运行时，不直接代表 P4 与 C5 的主联动边界

## 联调时为什么不要同时怀疑两边

如果网络失败，你很容易同时怀疑 C5、P4 和外部设备。但这样通常会把问题越查越大。

更稳的做法是按边界逐段验证：

1. C5 是否正常启动并提供预期能力
2. P4 是否正常启动并能进入控制台
3. P4 的网络配置是否有效
4. P4 的 UART 转发是否有效
5. 端到端消息是否真正通过

## 联调关注点

| 关注点 | 主要落在哪一侧 |
| --- | --- |
| 无线能力是否起来 | C5 |
| 运行时命令和参数是否生效 | P4 |
| 网络包是否进入桥接器 | P4 |
| 桥接后是否写到 UART | P4 |
| 上游配置是否被破坏 | C5 |

## 什么改动最容易影响联动

- 改变 C5 的传输相关配置
- 改变 P4 的网络默认行为
- 改变 P4 的 UART 默认行为
- 改变端到端数据进入桥接器的路径

做这些改动时，最好同步更新 [数据流文档](../interfaces/data-flow.md) 和相关图，否则后面的人很难快速判断问题是在边界前还是边界后。
