# 如何新增一个功能

适合谁看：
- 准备给 P4 或 C5 增加行为的人
- 想知道“应该先改哪里”而不是“哪里都想改”的人

读完会得到什么：
- 知道不同类型的需求通常落在哪些模块
- 知道新增功能时推荐的最小改动路径
- 知道如何减少破坏双端联调链路的风险

## 先判断你加的到底是什么功能

在这个项目里，“新增功能”差别很大。先分类，后动手。

### 运行时命令

例子：

- 新增一个 `net_*` 命令
- 新增一个 `uart_*` 命令

优先看：

- `console_app`
- `network_cmd` 或 `bridge_cmd`
- 对应业务模块

### P4 业务行为

例子：

- 调整网络默认行为
- 调整桥接逻辑
- 增加统计信息

优先看：

- `main.c`
- `network_app`
- `mavlink_bridge`

### C5 协处理器行为

例子：

- 调整传输配置
- 调整无线能力相关逻辑
- 调整上游能力暴露

优先看：

- `esp_hosted_coprocessor.c`
- `slave_control.*`
- `sdio_slave_api.*`
- `sdkconfig`

## 推荐的最小改动路径

1. 先把需求写成一句清楚的话
2. 判断它主要发生在 P4、C5，还是两端边界
3. 先改最接近需求本体的模块
4. 最后再补命令、文档和图

这个顺序的目的，是避免为了一个小需求而把很多外围模块一起改掉。

## 新增功能时最容易忘的三件事

### 1. 忘记同步文档

如果你加了命令、改了默认路径、改了 bring-up 顺序，就应该同步更新 `docs/`。

### 2. 忘记验证联动边界

只在本端通过并不代表双端一定没问题。尤其是碰到 P4 / C5 边界时，更要按数据流重新走一遍。

### 3. 忘记保持边界清晰

命令处理函数应该主要做解析。复杂业务逻辑应继续放在业务模块里。否则下次改行为时，会不知道该去哪一层修。

## 一个简单的判断法

如果你准备同时修改三个以上模块，先停一下，回答这两个问题：

1. 这次改动真正想改变的“可观察行为”是什么？
2. 哪个模块最靠近这个行为？

回答不出来时，通常说明需求边界还不够清楚，最好先补一段设计说明再改。
