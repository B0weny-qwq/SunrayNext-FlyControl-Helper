# 烧录速度矩阵实验报告

适合谁看：
- 想缩短日常烧录等待时间的开发者
- 想决定 `stub` 和 flash 波特率默认值的维护者

读完会得到什么：
- 知道 `stub on/off` 和不同波特率对 P4 / C5 的真实影响
- 知道这次实验里哪组配置最快
- 知道哪组配置更适合作为“日常默认值”

## 实验目标

验证以下两个因素对烧录总耗时的影响：

- `stub` 是否启用
- 串口烧录波特率

本次矩阵按每块板子各 6 组执行：

- `stub on` × `460800 / 921600 / 1500000`
- `stub off` × `460800 / 921600 / 1500000`

总计 12 组。

## 实验环境

- 日期：2026-04-14
- 主机：macOS
- P4 端口：`/dev/cu.usbmodem5B410437681`
- C5 端口：`/dev/cu.usbmodem5B410410081`
- P4 镜像：`firmware-p4/build/P4_WIFI_REMOTEv1.0.bin`
- C5 镜像：`firmware-c5/build/network_adapter.bin`
- 计时方式：`/usr/bin/time -p`
- 计时范围：单次完整 `write_flash` 命令，不含 `idf.py build`

## 实验方法

为了把 `stub` 和波特率拆开控制，本次没有直接使用 `idf.py flash`，而是直接调用 `esptool.py`。

P4 命令模板：

```bash
cd firmware-p4/build
python "$IDF_PATH/components/esptool_py/esptool/esptool.py" \
  --chip esp32p4 \
  -p /dev/cu.usbmodem5B410437681 \
  -b <baud> \
  --before default_reset \
  --after hard_reset \
  [--no-stub] \
  write_flash @flash_args
```

C5 命令模板：

```bash
cd firmware-c5/build
python "$IDF_PATH/components/esptool_py/esptool/esptool.py" \
  --chip esp32c5 \
  -p /dev/cu.usbmodem5B410410081 \
  -b <baud> \
  --before default_reset \
  --after hard_reset \
  [--no-stub] \
  write_flash @flash_args
```

说明：

- `stub on` 表示不传 `--no-stub`
- `stub off` 表示显式传 `--no-stub`
- 全部 12 组都成功完成，退出码均为 `0`

## 什么是 `stub`

`stub` 可以理解成“烧录阶段临时上传到芯片 RAM 里运行的一小段助手程序”。

`esptool.py` 在默认模式下，通常会先做两件事：

1. 通过 ROM 下载协议连上芯片
2. 把一段更高效的辅助程序上传到 RAM 并运行

日志里常见的：

```text
Uploading stub...
Running stub...
Stub running...
```

说的就是这个过程。

### `stub` 的作用

它不是你的业务固件，也不会常驻 Flash。它只在这次烧录会话里存在，主要作用是：

- 用更高效的方式擦写 Flash
- 提供比 ROM 原生下载协议更快的读写路径
- 让更高波特率下的烧录吞吐更容易发挥出来

所以大多数情况下：

- `stub on` 更快
- `stub off` 更保守，但通常更慢

### `--no-stub` 是什么意思

当命令里带上 `--no-stub` 时，`esptool.py` 不再上传这段 RAM 助手程序，而是直接使用芯片 ROM 自带的下载能力完成烧录。

这条路径的特点通常是：

- 兼容性更保守
- 行为更接近芯片最底层下载模式
- 速度一般明显慢于启用 `stub`

### 为什么本次实验里 `stub` 更快

这次矩阵里，两块板子都体现出同一个趋势：启用 `stub` 后，总耗时明显缩短。

这并不是因为业务镜像变小了，而是因为：

- Flash 擦写和传输过程更多地走了 RAM 里的高效助手
- 同样的串口波特率下，协议开销更低
- 高波特率时，吞吐更接近链路上限

### 什么时候可能会临时关闭 `stub`

虽然默认更推荐 `stub on`，但在下面这些场景里，`--no-stub` 仍然有价值：

- 某块板子在 `Uploading stub...` 阶段不稳定
- 某个下载器或接线组合下，`stub` 路径偶发失败
- 你想先确认最底层 ROM 下载链路本身是否正常

所以 `stub off` 更适合作为排障回退方案，而不是日常默认值。

## 观察结论

### 1. `stub` 是最大的速度杠杆

两块板子上，`stub on` 都明显快于 `stub off`。

- P4：`stub on + 921600` 相比 `stub off + 460800`，耗时从 `37.71s` 降到 `13.07s`
- C5：`stub on + 921600` 相比 `stub off + 460800`，耗时从 `33.58s` 降到 `10.56s`

这说明如果目标是缩短烧录时间，优先级最高的不是再抠 `flash_freq`，而是尽量走 `stub` 路径。

### 2. `460800 -> 921600` 提升非常明显

- P4：`16.19s -> 13.07s`
- C5：`15.89s -> 10.56s`

这一档提升很划算，而且本次实验中 2 块板子都稳定通过。

### 3. `921600 -> 1500000` 进入“边际收益变小”区间

- P4：`13.07s -> 12.99s`
- C5：`10.56s -> 9.65s`

P4 上几乎已经没有可感知收益。C5 上还有一点收益，但增量已经明显小于 `460800 -> 921600`。

### 4. C5 当前工程默认值偏保守

当前仓库里：

- `firmware-p4/sdkconfig` 没有启用 `CONFIG_ESPTOOLPY_NO_STUB`
- `firmware-c5/sdkconfig` 启用了 `CONFIG_ESPTOOLPY_NO_STUB=y`

但这次实测表明，在当前机器、当前下载器、当前接线条件下，C5 的 `stub on` 路径也是可用且更快的。

## 推荐配置

### 日常默认值

建议把“日常最合适配置”定为：

- `stub on`
- `baud = 921600`

理由：

- 相比 `460800`，提速非常明显
- 相比 `1500000`，速度损失很小
- 本次实验里 P4 / C5 都一次通过
- 更适合作为团队默认值，而不是只在单机单线材上追求极限

### 追求极限速度时

如果你确认当前下载器、线材、接触质量都稳定，可以临时使用：

- `stub on`
- `baud = 1500000`

适用场景：

- 连续反复烧录
- 单人本机开发
- 已经确认当前硬件链路不会因为高波特率变得不稳定

### 不建议作为默认值的组合

- `stub off + 460800`

这是本次实验里最慢的一档，只适合在 `stub` 路径异常时作为保守回退方案。

## 最终建议

如果后续要把仓库里的默认烧录策略往“更快但仍稳”的方向推进，优先推荐：

- P4：`stub on + 921600`
- C5：`stub on + 921600`

如果只是你个人在当前这台机器上追求最短时间，可以额外尝试：

- P4：`stub on + 1500000`
- C5：`stub on + 1500000`

但在把 `1500000` 设成团队默认值之前，最好再做至少 5 次重复实验，确认没有偶发失败或重试抖动。
